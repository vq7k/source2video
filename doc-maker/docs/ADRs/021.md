# ADR-021 · 无绕过 git 的 Web UI 上传 source（git + CLI 是 SOT；允许 git thin 前端）

**Status**: Accepted（2026-05-13 修正立场范围）

**Context**
- 用户问"source 怎么入仓 / 在哪里操作"。
- 候选有 Web UI 上传系统 / API endpoint / git + CLI / 基于 git 的 thin Web 前端。
- **修正动机**（2026-05-13）：[ADR-025](025.md) 提出业务层 console（L1），业务使用者上传素材需要 UI affordance。原"无 Web UI 上传"立场需要细化——**禁止的是"绕过 git 的 Web 上传"，不是"基于 git 的 thin 前端"**。

**Alternatives**
1. Web UI 上传 + 服务端存储 + 服务端版本化（绕过 git）。
2. API endpoint（程序化上传）+ 服务端存储。
3. **本地编辑器 + git + CLI**（采纳为 MVP 默认）。
4. **基于 git 的 thin Web 前端**：UI 后端调 `s2v cases lint && register && git commit`，**不绕 git**（采纳为业务层 console L1 的上传实现，见 [ADR-025](025.md)）。

**Decision**
- MVP 默认 source 入仓路径 = 本地编辑器 + `s2v cases lint/register` CLI + git commit。
- **禁止 1 / 2**——绕过 git 的 Web 上传 / API endpoint + 服务端存储都不许。
- **允许 4**——业务层 console（L1）的上传是"git thin 前端"：UI 后端写 staging 文件 → 自动调 lint → register → commit，保留 git / manifest / 审计 / CI 全套纪律。
- 何时该上 L1 业务层 console：业务侧第二轮启动后（见 [ADR-025](025.md)）。

**Rationale**
- git 已经是最好的 source 管理工具（版本化 / diff / 回滚 / PR review 都免费）。
- frozen 边界保护免费（git diff 自然检测 holdout 改动）。
- 审计追踪免费（git history 是天然 audit log）。
- CI 集成天然（source 入 git → CI 触发）。
- **核心立场不变**：禁止"重新造 git"——任何 Web UI 上传如果绕过 git 就违反本 ADR。
- **修正后允许的边界**：UI 作为 git/CLI 的薄壳前端，所有版本化 / 审计 / frozen 保护仍走 git——这种 UI 不"造 git"，只是给 git 加上传 affordance。差别在于"绕"还是"包"。
- [ADR-007](007.md) 精神（早上 framework 会被它的抽象绑住）+ [ADR-010](010.md) 精神（死掉的 UI 比没有 UI 更糟）继承。

**Consequences**
- [`../04-handbook.md`](../04-handbook.md) §2.0 Source 入仓 + 启动准备：明确 MVP 默认路径 = "本地编辑器 + git + CLI"。
- [ADR-025](025.md) 业务层 console 的"上传素材"按本 ADR 修正后立场实施：UI 后端是 git thin 前端，不绕 git，CI 检查仍能识别"通过 UI 上传的 commit"（commit 消息约定 `register case <id> (via business console)` 后缀）。
- 不混淆"无绕过 git 的 Web UI 上传"与"无 UI"——后者错。
  - L3 Per-Node Console 是 UI（[ADR-020](020.md)）
  - L2 Hub Console（[ADR-025](025.md) 提出，[`../06-ui-spec.md`](../06-ui-spec.md) 可承接） 是 UI
  - L1 Business Console（[ADR-025](025.md)）是 UI——其中包含本 ADR 修正后允许的 git thin 上传前端
- 第一版 MVP 仍省下"重新造 git"基础设施的实现 + 维护成本（L1 第二轮才上）。
