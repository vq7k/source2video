# ADR-012 · 白盒化可观测：Decision Trace + Eval Attribution

**Status**: Accepted

**Context**
- 老项目 ai-engineer-roadmap "全是人评估、没有 eval、没有可观测"。规模化后失去对 LLM 决策的可见性：**LLM 凭啥这么写？遵守哪条规则？违反哪条？** 全不知道。
- 仅有传统 LLM call trace（prompt / output / token）不够——这是"另一种黑盒"。

**Alternatives**
1. Langfuse trace 仅记 prompt / output（传统做法）。
2. **加 Decision Trace + Eval Attribution 两层白盒化字段**（采纳）。
3. 用外部评测平台（DeepEval / Phoenix）的内置 attribution。

**Decision**
- **Decision Trace**：每次 LLM call 必须记录 `rendered_prompt_diff / materials_injected / exemplars_used / llm_thinking_summary`。
- **Eval Attribution**：每条 rubric 维度 pass/fail 必须输出 `violated_rule_ref / violated_anchor / citation_in_artifact / judge_thinking`。
- 缺这些字段 = 该次执行直接判失败（框架级强约束，不是 lint）。

**Rationale**
- 解阶段 3 痛点 B（黑盒化）的唯一办法是"LLM 凭啥这么决策"可追溯。
- Attribution 是评估容量放大（[ADR-015](015.md) 配套）的前提：人不需要每条评估都看 LLM call，而是抽样审 attribution 看 judge 是否乱判。
- 缺字段判失败 = 强制框架开发者从 day 1 就做对，不允许"先没 attribution，后面补"——补不上。

**Consequences**
- 不变量 #3 + #4 落地（见 [`../03-invariants.md`](../03-invariants.md)）。
- Eval Stack 内部 judge 输出 schema 含 attribution 字段（[`../02-architecture.md`](../02-architecture.md) §2.3）。
- Observability 双层：传统 trace + Decision Trace（[`../02-architecture.md`](../02-architecture.md) §2.4）。
- US-04 升级为"白盒化可观测"，US-10 单独成 US（规则遵守可追溯）。
- [`../07-acceptance.md`](../07-acceptance.md) §5 MVP 出口判据 #4 #5 显式校验 attribution 与 Decision Trace 完整性。
