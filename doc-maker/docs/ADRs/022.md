# ADR-022 · 规则起源分类与 tag 体系（物料入口约束）

**Status**: Accepted

**Context**
- LLM 工程实践中，最大的"老化"来源不是新模型不够强，而是**老 scaffolding 还在叠加效果，反而拖累新模型**。
- 每加一条规则前必须回答：**这条规则该不该长期持有**？
- 不分类的后果：物料越积越厚，3 代模型后 prompt 变成考古层。

**核心判断**

这条规则要解决的问题对模型供应商而言是否"普适"：

- **所有调用方都希望模型这样做** → 在模型迭代路线上 → 12–18 个月内会被模型自身解决 → **临时补丁**
- **只有你或你这个行业才在意** → 业务侧永久负担 → **必须长期版本化维护**

**Alternatives**
1. 不分类，全堆一起（早期错误，老化主因）。
2. 只在 prompt 注释里说明"这条以后可能删"（缺机械执行力）。
3. **强制 tag + 物理隔离 + 模型升级强制重跑回归**（采纳）。

**Decision**

**四个测试**判断 tag：

| 测试 | 问什么 | 处置 |
|---|---|---|
| 普适性 | 所有 LLM 调用方都撞到吗？ | 是 → `capability_gap` |
| 私有信息 | 依赖只有这边知道的信息？ | 是 → `business_policy` |
| 趋势 | 老 prompt 在最近 2–3 代模型盲测，行为变了吗？ | 变好 → `capability_gap`；没变 → `business_policy` |
| 后果不对称 | 模型能做对但代价能承受吗？ | 不能 → `business_policy`（即使能力够也强制显式校验） |

**四个 tag**（强制 4 选 1）：

- `capability_gap` — 模型能力不足的临时补丁（**模型升级强制重跑回归**）
- `business_policy` — 业务/品牌/合规/私有信息（长期持有 + 版本化 + 单测）
- `channel_constraint` — 渠道/格式约束（跟外部接口同步）
- `integration_glue` — 系统集成胶水（跟集成方一起 review）

**物理隔离两层**：

```
nodes/<name>/prompts/
├── _model_adapter/      ← capability_gap（易耗品）
└── _business_rules/     ← 其它三类（长期资产）
```

**升级期强制重跑**：模型升级后 CI 自动跑所有 `capability_gap` 规则的"无该规则" vs "有该规则"对照，新模型已稳 → 标 obsolete_candidate → 人工 review 删除。

**灰区 brand voice 类**：既属普遍写作能力又涉私有偏好——留专门升级路径（重 prompt → few-shot → fine-tune），metadata 标 `evolution_stage`。

**Rationale**
- 不分类 = 物料无方向地堆积，3 代模型后无人敢动。
- 物理隔离让"模型升级时该删哪些"在文件系统层就是显式的，不靠开发者记忆。
- 强制 CI 校验 = 不依赖自觉，机械执行。
- 大多团队真实痛点不是新模型不够强，而是老 scaffolding 拖累——这条 ADR 直接对症。

**Consequences**
- 不变量 #12a / #12b / #12c 落地（见 [`../03-invariants.md`](../03-invariants.md)）。
- [`../02-architecture.md`](../02-architecture.md) §2.2.3 落实四个测试 + 两层物理隔离 + tag schema。
- [`../04-handbook.md`](../04-handbook.md) 用户加新规则时强制选 tag（流程纪律）。
- 模型升级 = 框架重大事件——触发 CI 全量回归 + capability_gap 清理。
