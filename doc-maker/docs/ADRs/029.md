# ADR-029 · Framework Core 可迁移内核与 Langfuse Lens 边界

## 背景

当前 Writing Production 原型已经有 L1 业务工作台、local JSON run store、最小 `TraceSink` 和 deterministic eval。实现过程中出现了一个偏移：`/framework` 被做成了聚合诊断页，容易变成自研日志中台，并且把业务概念和框架概念混在一起。

这与既有决策冲突：

- ADR-007：第一版不上编排 framework，但 Langfuse 必须早接。
- ADR-010：Per-node UI，禁止跨节点聚合。
- ADR-012：LLM call 必须 Decision Trace；Eval 必须 Attribution。
- ADR-025：L1 业务层与 L2/L3 框架层解耦。
- ADR-028：`TraceSink` 是 Langfuse 适配边界，不自研观测平台。

## 决策

采纳以下边界：

1. **Framework Core 是业务无关内核**。
   它只定义 `NodeSpec / NodeRun / ArtifactRef / EvalRun / FeedbackEvent / TraceSink / ScoreSink`。

2. **Writing 只通过 adapter 接入**。
   `writing-*` 可以依赖 `workflow-core`，但 `workflow-core` 不能 import `writing-*`。

3. **Langfuse 是观测 SOT**。
   Trace、Observation、Score、token、latency、error 进入 Langfuse；local JSON 只做 fallback 和开发态缓存。

4. **`/framework` 改为 Langfuse Lens**。
   它不是日志中台，不平铺 run。它只根据 `runId / nodeRunId / artifactId / round / candidateId` 展示单节点诊断，并提供 Langfuse 深链。

5. **Eval 节点先接 Scores，不先做评测平台**。
   当前 deterministic eval 先写 `EvalRun` 与 Langfuse Scores；LLM judge、dataset、experiment 后置。

完整实现设计见 [`../framework-core.md`](../framework-core.md)。

## 后果

- `FrameworkNodeType` 这类业务枚举需要迁移为通用 `nodeId: string`。
- `.writing-runs` 保留为 L1 业务 store，不作为框架层 SOT。
- 新增业务节点时，只需要注册 `NodeSpec` 和业务 adapter，不改框架内核。
- L1 不展示 raw trace；只从业务节点跳转到带条件的 Lens。
- `/framework` 当前页面需要收敛，不能继续扩成可观测平台。

## 实施顺序

1. 文档与命名收口。
2. 抽出 `workflow-core` contract。
3. 增加 `LangfuseTraceSink` 与 `ScoreSink`。
4. writing adapter 接入 core contract。
5. L1 增加节点级执行详情入口。
6. `/framework` 改为条件化 Langfuse Lens。
