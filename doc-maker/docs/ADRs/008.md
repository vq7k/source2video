# ADR-008 · 反馈流转与物料版本化

**Status**: Accepted

**Context**
反馈到初始物料是整个系统最容易死的环节。无纪律 → 3 个月后 prompt 变成"谁也不敢动的诅咒"。

**Decision**

**反馈落点 5 层**（同一条反馈可能落到任一层，判断在哪一层修是关键）：

| 层级 | 影响范围 | 风险 |
|---|---|---|
| 系列 style guide / 术语表 | 全系列 | 高 |
| 每步 prompt | 所有未来 plan | 高 |
| Schema / 字段定义 | 所有 plan + 下游 | 高 |
| Judge rubric | 所有 plan 评估方向 | 中 |
| Few-shot exemplars | 定向影响 | 低 |
| 单期 plan | 仅一期 | 无 |

**反馈格式强制结构化**（每条必填 `likely_cause`，是聚类归因的命根子）。

**工作流（强制）**
1. 收集（不立即触发改动）
2. 每周/两周 triage 聚类
3. 归因到具体层
4. 改动前 baseline（regression 集跑当前 prompt）
5. 改动后 regression 验证
6. 版本号 + 改动日志

**物料版本化（必须）**

| 物料 | 方式 |
|---|---|
| Prompts | Git + 语义化版本 |
| Schemas | Pydantic + version field |
| Style guide / 术语表 | Git YAML + 版本号 |
| Rubrics | 同上 |
| Exemplars | DB / git，每条有 id + 加入日期 |
| **Plan artifact** | **内嵌所有上述物料的版本号** |

最后一条是命根子：任何产物都能反查"配方"。

**Regression 集（必须建）**
20–30 期已做完且满意视频 → 反推 plan → frozen regression set。任何 prompt / rubric / style guide 改动先跑这个集，通过才上线。

**Rationale**
- 没有 regression 集，prompt 迭代就是赌博。
- 反馈不分层归因 → 一定会"修一个坑挖三个新坑"。
