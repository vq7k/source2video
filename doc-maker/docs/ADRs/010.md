# ADR-010 · 管理界面：Per-node UI，禁止聚合

**Status**: Accepted

**Context**
管理界面如果集中成"全流程 console"，会在管理层重新引入所有节点之间的耦合——架构白拆。

**Decision**
**每节点配独立 UI**，只覆盖本节点：
- 本节点输出 artifact（结构化渲染）
- 本节点输入物料版本
- 本节点反馈表单
- 本节点 eval 指标
- 本节点 rerun 按钮

**跨节点导航**靠 artifact ID 超链接跳转，**不做聚合视图**。每个 console 独立部署、独立可关停。

**操作分类三档**

| 档 | 处置 |
|---|---|
| 必须自建（高频高密度） | per-node console |
| 用现成工具 | Langfuse / Phoenix / Git / Notion / Linear / Argilla |
| 留在 CLI | 批量 rerun / 物料 promote / schema migration / 指标导出 |

**禁忌**
- 不要建自己用不上的功能（"为新人 / 老板"建的页面通常没人点）。
- 架构图用 Mermaid 手画放 Notion，不要动态生成。
- 死掉的 UI 比没有 UI 更糟。

**Rationale**
- 物理解耦 + UI 解耦 = 真解耦。
- 谁拥有节点谁拥有该节点的 prompt / rubric / UI / 文档，一脉相承。
- 复杂系统的人脑管理，主要靠 **文档纪律 + 命名 + 版本号**，UI 是锦上添花。

**Consequences**
- 文档目录降到节点级：`nodes/<name>/{README, prompts/, rubric.yaml, eval/, ui/}`，仅一份 `index.md` 列节点与连接。
- 第一周建的是 **Plan 节点 console**（100–200 行 Streamlit），不是"期评审 console"。
