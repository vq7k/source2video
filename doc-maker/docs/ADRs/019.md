# ADR-019 · 节点级目录结构（nodes/<name>/ 全套就近）

**Status**: Accepted

**Context**
- [ADR-010](010.md) 提出节点级文档目录，但只覆盖"文档 + UI"。
- 早期物料集中存放在 `fixtures/materials/<node>/v1.0/`、代码在 `nodes/`、UI 在第三处——**耦合在文件树拓扑里隐藏**。
- 改节点要跳 4 处，违反"per-node 责任绑定"精神。

**Alternatives**
1. 中央集中（fixtures/ + nodes/ + ui/ 分散）（早期错误）。
2. **节点就近**：`nodes/<name>/{README, node.py, prompts/, rubrics/, schemas/, ui/, eval/}`（采纳）。
3. 单一仓库平铺（无目录分层）。

**Decision**
- 每个节点的物料 / 代码 / UI / eval / 反馈 / 文档**全住在 `nodes/<name>/` 一个目录**。
- `fixtures/` 仅保留**跨节点共享的输入数据**（cases、regression baseline）。
- `index.md` 仅列节点和连接，不重复细节。

**Rationale**
- 不变量 #8（UI 不聚合）在 UI 层不够，必须延伸到**文件系统拓扑**。
- 谁拥有节点 → 谁拥有 prompt / rubric / UI / 文档（一脉相承）。
- 改节点只在一个目录里，新人接节点看一个目录就齐。
- **解耦在文件树拓扑里显式**，不靠开发者自觉。

**Consequences**
- [`../02-architecture.md`](../02-architecture.md) §2.6.1 落实节点级目录结构。
- [`../07-acceptance.md`](../07-acceptance.md) §3.2 fixtures 目录改为"节点就近"。
- [`../06-ui-spec.md`](../06-ui-spec.md) §4 index.md 是文档不是 console，全系统索引由它承担。
- 实施时：每个 `nodes/<name>/` 目录是一个完整 unit，可单独 review、删除、迁移。
