# ADR-028 · 当前主线从讲解文档包抽象上移为 Writing Production

## 状态

采纳。

## 背景

上一轮业务产品原型把 doc-maker 定位为“源材料 -> 结构化讲解文档包”的工作台。UI mock 后暴露出产品抽象层级偏低：真实问题不是只缺一个文档包生成器，而是用户面对混乱素材、外部写法参考、视频转录稿和不稳定审美时，无法稳定地产出可比较、可评分、可沉淀的文本。

因此，产品需要从“产物类型驱动”抽象上移为“文本生产闭环驱动”。原讲解文档包设计不删除，保留为默认 `Output Profile`。

## 决策

doc-maker 当前主线调整为 **Writing Job 文本生产系统**：

```text
Raw Input
  -> Start Context（Manual Rule Scope）
  -> Quick Intake + optional Reference Paste
  -> Writing Rule Scope
  -> Baseline Stack
  -> Job Spec + Output Contract
  -> Precheck
  -> Content Brief + Writing Rules Candidate + Eval Profile + Risk List
  -> Candidate Run
  -> Auto Eval + Human Selection Feedback
  -> Feedback Ledger + Rule Patch Drafts
  -> Next Generation Run + Rule Snapshot
  -> Finalize Export
```

讲解文档包保留为首个输出场景，不再锁死产品本质。

## 理由

1. 输入天然动态，必须先用 Job Spec 降低复杂性。
2. Precheck 不是简单清洗，而是把素材、写法、评审规则编译成固定可审结构。
3. 第一轮 Eval Profile 不能为空，否则多篇候选无法比较，人工反馈也无法归因。
4. Writing Rules Candidate 是可沉淀规则资产；Published Skill Package 才是标准能力包，不能由一篇参考文章或一次主管反馈直接发布。
5. 产品价值在生成、评分、局部反馈、规则迭代和定稿导出，不在富文本编辑。
6. 反馈必须先进入账本和规则草稿，不能直接改写旧候选。
7. 规则草稿和 active rules 必须有上限，避免反馈堆积污染后续生成。
8. 初始化层先做 Manual Rule Scope，不做 Source Store / RAG；否则会提前引入检索、分块、权限、provenance 和额外 eval 面。

## 后果

- L1 首页从 Episode 列表改为 Writing Job 工作台。
- `business-console.md`、`07-acceptance.md`、`ui/README.md` 的当前主线改为 Writing Production。
- L2/L3 诊断层继续保留，但对业务用户默认不可见。
- 旧 Episode / 文档包页面保留为默认 Output Profile 的查看入口和诊断 mock。
- 技术导出（Codex/Claude `SKILL.md`）降级为高级动作，只能从 Published Skill Package 生成。
- 当前 baseline 原型已从本地行为 mock 推进到可执行闭环：Writing Rule Scope、Precheck Normalization、Candidate Generation、Feedback Reasoning、RulePatch Compilation 可接真实 runtime；Candidate Auto Eval 仍是 deterministic core eval；不接 RAG 或 Skill Package 发布仓库。
- 下一步实施范围收敛为 Quick Intake + Reference Paste -> Writing Rule Scope -> Baseline Stack，不接 Source Store / RAG。

## 2026-05-16 收口补充

暂停新增 Candidate guard / regression fixtures 等扩展能力。当前优先级改为边界收口：

- L1 只保留业务状态和最小反馈路径。
- `/framework` 只作为真实 run 的诊断入口，不演化成新的业务工作台。
- `TraceSink` 是 Langfuse 适配边界，不自研观测平台。
- `writing-runtime.ts` 只维护业务状态流；framework trace 组装独立到 `writing-framework-trace.ts`。
