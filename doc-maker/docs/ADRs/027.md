# ADR-027 · 当前主线从 framework-first MVP 调整为业务产品原型

**Status**: Accepted（2026-05-15）

> 2026-05-16 更新：本 ADR 仍保留“业务产品原型成为当前主线”的决策；产品抽象已由 [ADR-028](028.md) 进一步上移为 Writing Production。讲解文档包不删除，作为默认 Output Profile 保留。

**Context**

- 原计划以 ToyNode framework MVP 为第一轮主线，纪律是“测框架，不测业务”（[ADR-011](011.md)）。
- 推进 UI mock 后，实际学习重点发生变化：业务侧产品界面暴露了更多关键问题，包括 Episode 状态模型、业务验收、产出边界、warn/fail handoff、L1 与 L3 的信息边界。
- 用户确认产品抽象为：**doc-maker 是把一组源材料转换成可审阅、可追溯、可迭代的结构化讲解文档包的生成工作台**。
- 当前不锁死具体业务域；“讲解型内容”是首个落地场景，首版文档包为 `plan / scripts / shots / visual_spec / qa_report`。

**Alternatives**

1. **回到 ToyNode-only framework MVP**
   好处是边界干净；问题是会浪费 UI mock 已经暴露出的业务产品学习。
2. **直接把完整 UI mock 当成已验证产品**
   好处是推进快；问题是 Plan/Shot/QA backend 尚未真实接线，容易误判 ready 状态。
3. **业务产品原型成为当前主线，L2/L3 标为诊断故事板**（采纳）
   保留产品学习，同时不伪装 backend readiness。

**Decision**

当前工作主线调整为 **business prototype**：

- L1 Business Console 与 L1 Output Viewer 是当前产品契约。
- L2/L3 仍保留为诊断层，但在真实 artifact 接入前必须标注为诊断故事板 / mock-only。
- Framework ToyNode MVP 仍作为历史基线和诊断闭环参考，不再覆盖当前业务产品原型验收。
- 当前业务产品原型的验收以 [business-console.md](../business-console.md) 与 OpenSpec `promote-business-prototype` 为准。

**Rationale**

- 业务产品原型现在回答的是“doc-maker 对使用者有什么价值”，不是“框架机制是否全部跑通”。
- L1 必须先成为一个独立产品面：上传 source、查看状态、验收文档包、在问题时 handoff 到诊断。
- L3 仍重要，但它服务的是可观测 eval 闭环：trace / eval / feedback / regression / version bump。
- 将 L2/L3 标为故事板，可以继续进行产品探索，同时避免把 mock 页面误读成生产实现。

**Consequences**

- [`../business-console.md`](../business-console.md) 从 `_future/` 提升为当前业务产品原型 SOT。
- [`../07-acceptance.md`](../07-acceptance.md) 必须区分业务产品原型验收与 framework ToyNode MVP 验收。
- [`../06-ui-spec.md`](../06-ui-spec.md) 继续承担 L2/L3 诊断层 UI 规范，不再把 L1 描述为第二轮才动。
- [`../../ui/README.md`](../../ui/README.md) 必须标注路由是产品契约、诊断故事板还是 mock-only。
- 后续实现优先修 L1 产品状态模型和业务产出边界，再处理诊断层接线。

**不变**

- [ADR-021](021.md) 仍要求上传 / rerun 走 git-thin，不绕过 git。
- [ADR-025](025.md) 的 L1/L2/L3 解耦仍成立。
- [ADR-020](020.md) 仍适用于 L2/L3：诊断层 UI 是作者活体地图。
- [ADR-011](011.md) 不删除，但降级为 framework-first 历史基线，不再定义当前主线。
