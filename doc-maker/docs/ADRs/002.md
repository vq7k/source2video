# ADR-002 · 节点粒度：变更频率 + 失败异质性驱动，粒度不追求均匀

**Status**: Accepted

**Context**
- 用户原维度："认知成本 vs 流程复杂度"。
- 实际上还有更强的两轴未识别：变更频率、失败异质性。

**Decision**
采用五轴决策（按重要性排序）：

| 轴 | 含义 |
|---|---|
| 变更频率不一致 | "几乎不变" 与 "经常重做" 混在一起 → 拆 |
| 失败异质性 | 不同失败原因、不同 retry 策略 → 拆 |
| IO 可独立定义 | 不能干净定义 → 不许拆（共享状态拖死） |
| 认知成本 | 仅在前三条成立时作为加分项 |
| 流程复杂度 | 常驻税（schema/eval/retry/缓存键/学习成本） |

**操作判断题（四个全过再拆）**
1. 会单独重做这部分吗？
2. 失败模式和邻居不一样吗？
3. 输入输出能干净定义吗？
4. 拆了之后 eval / debug / 缓存真的变简单吗？

**反直觉原则：粒度不该均匀**。稳定 + 高成功率 → 粗；高频重做 / 高失败率 → 细。

**Rationale**
- 粒度错误代价不对称：太粗只是烦（全量重做、随时可拆），太细是系统性问题（常驻税、合并难度远大于拆分）。
- 共享 immutable 输入不构成耦合，不该成为拒绝细拆的理由（见 [ADR-003](003.md)）。

**Consequences**
- 启用"敢拆也敢合"的双向操作纪律，防止系统越积越碎。
- 触发器：出现 3 次以上"只想改 sub-part X" → 拆；两节点总是一起改 / 一起失败 → 合并。
