# ADR-013 · 业务侧治理实践不在框架内核

**Status**: Accepted

**Context**
- 早期 Eval Stack 模块图把 评审校准 / 对照样本 / 多评审 / 相关性校验全塞进框架。
- 这些条目性质 ≠ 框架自动能力——它们是**人怎么用框架的运营纪律**，跟"业务做什么、几个评审者、多久跑一次"高度耦合。
- 读者会误以为 MVP 必须交付这些。

**Alternatives**
1. 保留在 Eval Stack 内部（早期错误做法）。
2. **移出框架内核，独立 §2a "业务侧治理实践" 章节**（采纳）。
3. 完全删掉，不在任何文档提。

**Decision**
- 把 评审校准 / 对照样本 / 多评审 / 相关性校验 / rubric 反演 5 件治理实践从 Eval Stack 模块图移出。
- 框架可提供 CLI 工具支持（如 `s2v eval calibrate` / `s2v eval correlate`），但**触发条件、人工流程、阈值校准由业务侧决定**。
- 标 P2 / 第二轮+ 引入。

**Rationale**
- 治理实践跟业务运营高度耦合，硬塞框架 = 框架变成业务运营管道。
- "MVP 必须交付" 与 "远期可选" 必须分清，否则 MVP 范围爆炸。
- 第一版只有一个用户（作者本人）+ 一个节点（ToyNode），治理实践对应的协作场景根本不存在。

**Consequences**
- [`../02-architecture.md`](../02-architecture.md) §2a 新章节明确边界。
- 不变量 #10 改写："相关性校验"从框架不变量移到治理实践（保留为强烈推荐，不强制），见 [`../03-invariants.md`](../03-invariants.md)。
- [`../07-acceptance.md`](../07-acceptance.md) US-14 标 P2；MVP 不交付任何治理。
- [ADR-009](009.md) 内容仍然适用，但作为"业务侧第二轮+ 治理纪律"，不作为框架第一版必须做的事。
