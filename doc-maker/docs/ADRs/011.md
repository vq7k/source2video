# ADR-011 · 第一轮纪律：测框架，不测业务

**Status**: Accepted (2026-05-12 重构期立)

**Context**
- 框架重构期反复混淆"验框架能力"与"验业务质量"。
- 早期版本把 Plan 节点作为 dogfood、把 KNN 反推 plan 作为 fixtures、把"plan 一次过率"作为 MVP 出口指标——这些全是业务测试，不是框架测试。

**Alternatives**
1. 用真实业务节点（Plan / Shot）+ 真实业务数据做 dogfood。
2. **人造 ToyNode + synthetic fixtures**（采纳）。
3. 完全不 dogfood，纯 unit test。

**Decision**
- 第一轮 MVP 只交付框架能力 + 一个人造 ToyNode + synthetic fixtures。
- **业务节点（Plan / ShotExecutionNode）的开发与验收全部延后到第二轮**。
- MVP 出口判据 = 8 条框架能力（trace 完整 / attribution 完整 / auto-judge / regression / console 等），**零业务质量指标**。

**Rationale**
- 隔离性：用真实业务数据，框架 bug 跟业务 bug 不可区分。
- 可控性：业务数据有质量噪声 + 上下文耦合，框架测试要 controlled input。
- 稳定性：业务数据可能改，synthetic 是 fixture 永不变。
- 纪律一致：不混淆 = 后续每一轮决策都有清晰边界。

**Consequences**
- US-09 dogfood 节点 = ToyNode "MarkdownPointsExtractor"，跟讲解视频无关。
- fixtures 全 synthetic（见 [ADR-018](018.md)）。
- "plan 一次过率 / 可执行率 / rule_audit" 全部从 MVP 出口移除，作为业务副产物记录但不进 gate。
- `../_future/business-design.md` 重新定位为业务侧第二轮工作，不在第一轮范围。
