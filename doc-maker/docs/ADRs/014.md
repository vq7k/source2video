# ADR-014 · Materials 演化三动作（行为变化归位规则）

**Status**: Accepted

**Context**
- 节点行为需要变化时（审美收紧 / 输出类型新增 / IO 契约改了），不同性质的变化该走不同路径。
- 没有归位规则 → 任何变化都可能演化成"改节点代码"，节点跟主观偏好反复改，永远稳不下来。

**Alternatives**
1. 所有行为变化都改节点代码（最差，老项目模式）。
2. **三动作分流**：审美变 → 改 Materials；类型变 → 新增 Materials；IO 契约变 → 改节点代码（采纳）。
3. 完全不区分，全靠开发者自觉。

**Decision**
按变化性质分流：

| 触发 | 动作 | 例子 |
|---|---|---|
| 审美变（同类型同任务，标准改了） | **改 Materials**（bump 版本号） | rubric 收紧 anchor；prompt 加一句 |
| 类型变（任务/受众/输出形态新增，旧的仍要保留） | **新增 Materials**（v1/v2 并存或新目录） | 同节点服务"视频版" + "播客版" |
| 节点 IO 契约变 | **改节点代码**（升 Node.version） | artifact 新增字段；输入新增引用 |

**Rationale**
- 节点代码只对"契约"负责，不对"风格 / 标准 / 审美"负责。审美变化绝不动节点代码。
- 默认动作：先想能否落到 Materials；想不到才动节点代码——这是稳定性的保险。

**Consequences**
- 不变量 #11 落地（见 [`../03-invariants.md`](../03-invariants.md)）。
- [`../02-architecture.md`](../02-architecture.md) §2.2.1 演化三动作小节明确规则。
- "改节点代码" 成为代价最高的动作，纪律上压低发生频率。
