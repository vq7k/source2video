# 轻量文本生成台 L1 设计原则

> 时间：2026-05-21  
> 适用范围：`/writing` 业务入口。它是轻量 L1，不替换 `/` 综合工作台，也不替换 `/framework` 观测 Lens。  
> 配套调研：[`lightweight-writing-workbench-research.md`](./lightweight-writing-workbench-research.md)。

## 1. 产品定位

轻量 L1 不是任务管理系统、不是 Chat 客户端，也不是规则管理后台。

一句话：

> 用户输入一句文本任务或一段材料，系统生成 3 个可比较版本；用户用极少反馈推动多轮收敛，最后复制或标记一个可用版本。

当前设计纪律：

1. **开箱即用**：不要求用户先建规则、模板、项目或工作流。
2. **只做文本生成**：不预设“写作”使用方式，输入可以是问题、说明、方案、提示词或材料整理。
3. **先出结果**：第一屏聚焦一个输入框和生成动作，不先展示 Precheck 表单。
4. **过程轻可见**：生成时用自然语言状态提示，不暴露 G1/E1、Node、prompt payload、token 等内部词。
5. **规则后置**：临时规则在 Topic 内滚动；可复用规则包属于 L2/L3 或后续能力，不进入 L1 默认心智。

## 2. 核心对象

当前最小模型：

```text
一次页面使用
  ├─ Topic A：一个文本任务
  │   ├─ Round 1：3 个候选 + eval
  │   └─ Round 2：基于反馈再生成 3 个候选
  └─ Topic B：另一个文本任务
```

| 概念 | 用户理解 | 系统含义 |
|---|---|---|
| 本次 | 当前浏览器页签里的工作集合 | `sessionStorage` 保存 Topic id；关闭页签自然清空 |
| 以往 | 历史 Topic | 持久化 run 记录；默认折叠，点开后加入本次 |
| Topic | 一个文本任务 | 边界锁，限制多轮迭代不发散 |
| Round | 一次生成尝试 | 当前由 `WritingRunRecord` 承载 |
| Candidate | 一轮里的 3 个候选版本 | 每个候选独立生成，并独立 eval |
| Eval | 候选评分节点 | 评分不由生成节点产出 |

实现边界：

- Topic 仍以 run 为持久化单元，不新增复杂 Session 对象。
- “本次”不是历史会话导入；只是当前页签的工作视野。
- 历史 Topic 数据不丢，但默认不占用心智。

## 3. 主流程

轻量 L1 的主循环：

```text
输入文本任务
  -> 自动理解输入边界
  -> 生成 3 个候选版本
  -> 每个候选独立 eval
  -> 展示当前最高 eval 版本与其余折叠版本
  -> 用户复制 / 标为最终 / 再来一轮
```

注意：

- “再来一轮”不是完整 Chat 历史续写，而是读取原始 Topic、当前规则快照、已选版本、用户补充要求和反馈方向。
- 没有反馈方向时允许再来一轮，系统按当前边界重新生成；UI 只做轻提示，不强制拦截。
- 第 5 轮后只显示事实提示：`已到第 5 轮，可回看前几轮差异。` 不判断“已经没进步”。

## 4. 输入原则

最小 Topic：

> 一句能让人回答“这要写什么”的话。

合法输入可以很短：

| 输入 | 是否可用 | 原因 |
|---|---|---|
| `什么是 Mamba 架构？` | 可用 | 问题明确 |
| `用 300 字解释：为什么 RAG 不适合实时数据？` | 可用 | 任务和长度明确 |
| `根据下面内容生成一段提示词：...` | 可用 | 产物类型明确 |
| `AI` | 不稳定但允许 | 过短，系统可以生成，但结果发散风险高 |

L1 不做强制拦截。后续可以增加“自动优化输入”按钮，但当前暂不加，避免工具变重。

## 5. 候选展示

当前实现：

1. 最新或选中 Round 中，默认展开 eval 最高候选。
2. 其余候选默认折叠，显示标题、Eval、强项和一句信号。
3. 分数用于解释推荐依据，但不作为用户必须接受的结论。
4. 历史 Round 可切换，旧候选不会被新规则改写。

已确认但暂缓的改造：

- P4：真正的“平权候选”不是删 badge，而是让 3 个候选拥有可比较的角度标签和等权入口。
- 不采用 3 篇长文全文横向平铺；长文本会降低阅读效率。
- 后续更合理的形态是“3 个等权候选入口 + 单正文阅读区”。

## 6. 反馈与完成

L1 反馈必须轻。

当前动作：

| 动作 | 用户含义 | 系统处理 |
|---|---|---|
| 复制文本 | 直接拿走当前版本 | 不改变 run 状态 |
| 标为最终 | 这个版本可作为本轮结果 | 写入 `finalizedCandidateId`，保留服务端 finalized 能力 |
| 再来一轮 | 当前还不够好 | 记录方向/补充要求，生成下一轮 3 个候选 |

反馈方向：

- 保留方向
- 更具体
- 调结构
- 调语气
- 换个角度

这些标签是快捷信号，不是强制评分表；文本补充框与标签互补。补充框用于硬事实或具体要求，标签用于快速表达方向。

## 7. 规则策略

当前 L1 默认没有正式规则。

| 场景 | 规则行为 |
|---|---|
| 新 Topic 无规则包 | LLM 生成临时边界和候选规则 |
| 进入下一轮 | 继承当前 Topic 的临时上下文、反馈和规则补丁 |
| 选择规则包 | 后续能力；选择后应停止自动生成初始规则 |
| 导出规则资产 | 不在 L1 默认流程内，后续交给 L2/L3 的规则提取器或 Skill Package 工具 |

关键纪律：

1. 预设流程，不预设具体内容规则。
2. LLM 生成的内容必须可追溯、可打标。
3. 临时规则只影响当前 Topic，不自动污染跨 Topic 规则库。
4. 规则更新不改写旧候选，只影响下一轮。

## 8. 页面结构

```text
/writing
  Header：doc-marker / 文本生成 / 新建 / 全貌 / 观测
  左侧：本次 Topic + 折叠以往 Topic
  中央：输入文本 / 上下文折叠条 / 历史结果 / 当前候选 / 反馈
```

当前删除或降级的元素：

- 删除右侧 320px ContextPanel。
- 删除“下一步”教学卡。
- 删除 8 个内部 pipeline 步骤格子。
- 删除“先出结果”等价值承诺 badge。
- Header 不再显示 `L1`、`旧版`、`轻量写作台`。

保留的轻入口：

- `全貌`：进入综合工作台。
- `观测`：进入 Framework / Langfuse Lens，只在有 active Topic 时可用。

## 9. 验收标准

| 标准 | 要求 |
|---|---|
| 5 秒可理解 | 用户看到一个输入框，知道可以生成文本结果 |
| 不泄漏内部词 | 首屏不出现 G1/E1、Rule Snapshot、prompt payload、Node |
| 多轮清楚 | 用户知道仍在同一个 Topic 下迭代，旧轮次可回看 |
| 历史不打扰 | 以往记录默认折叠；本次只显示当前页签 Topic |
| 出口可见 | 复制文本、标为最终、再来一轮都可直接找到 |
| 不污染数据 | 新规则/反馈不改写旧候选 |
